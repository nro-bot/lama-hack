Data

Three datasets, one skill

Dataset 1

Individual notes

One note per episode — each demonstration strikes a single labeled bar. Published as ArjunPrasaath/training_xylophone on HuggingFace.

120 episodes
Dataset 2

All notes

Longer episodes covering every bar in sequence, giving the policy context on transitions between bars.

10 episodes
Dataset 3

Combined

The union of both — per-note precision from dataset 1, whole-instrument coverage from dataset 2.

130 episodes

Every episode carries one of eight fixed instruction strings (e.g. Hitting note D (orange)) — the exact labels the sequencer replays at inference time.

Training

Imitation learning, run twice

Platform A

New Theory training platform

Managed fine-tuning of the MolmoAct2 base model on the LeRobot datasets — upload data, pick the base, launch.

Platform B

Modal

The same recipe run as our own GPU job on Modal, for full control over the training loop and checkpointing.

  • Behavior cloning on teleop demos — the policy imitates the human strikes; no reward signal, no simulator.
  • 8,000 training steps — per run, from the MolmoAct2 base.
  • Output: ArjunPrasaath/play_xylophone_100 — a public LeRobot-format checkpoint on HuggingFace, the one this repo is tuned for.

Inference · this repo

Two ways to serve, one client

Option A · bring your checkpoint

HuggingFace → Modal

modal deploy server/modal_ws.py loads the HF checkpoint onto a GPU and exposes a WebSocket endpoint. Point NT_INFERENCE_URL at it. Works for any LeRobot-format MolmoAct2 fine-tune.

Option B · nothing to deploy

New Theory inference API

newt login once, then run.py --backend newt --model <tag> runs against a model in New Theory's registry — no server, no GPU of ours.

--backend auto (the default) picks A when NT_INFERENCE_URL is set, B otherwise — after setup, plain run.py --song ... just works.

Runtime

A single-note policy that plays songs

  • One WebSocket, whole song — the newt SDK rebuilds the prompt from the observation dict every frame, so the sequencer swaps the instruction under one persistent run() call. No audible reconnect gap between notes.
  • Guarded vocabulary — notes.py is the chokepoint: regex parsing, with a Claude fallback constrained to an enum of the eight trained keys. Nothing untrained ever reaches the model.
  • Verified before hardware — full smoke test against the deployed server: correct (30, 6) action chunks, all 8 labels accepted, different notes produce different actions.
